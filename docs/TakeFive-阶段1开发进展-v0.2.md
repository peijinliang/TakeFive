## 1. 本轮结论

TakeFive 已从“规则和存储原型”推进到真实可运行的固定时间与一次性提醒闭环：

```text
SQLite 已启用规则
  -> 生成 (since, now] 到期候选
  -> 查询单条/全局暂停与系统会话状态
  -> 原子认领 occurrence
  -> 持久化策略决定
  -> Tauri 系统通知
  -> presented / delivery_failed
```

主窗口关闭后 Rust 主进程和调度循环继续运行。创建、删除、启动、定时到期、唤醒、解锁、系统时间变化和时区变化都会触发重新对账。

当前全仓共 **84 项 Rust 测试**，全部通过；TypeScript 类型检查、Vite 生产构建和严格 Clippy 检查通过。

## 2. 本轮新增能力

### 2.1 SQLite CandidateSource

- 从 `list_scheduled_enabled` 读取当前启用且未删除的提醒。
- 反序列化固定时间与一次性规则，严格生成 `(since, now]` 候选。
- 候选包含 reminder revision、稳定 occurrence key、UTC/本地时间、时区和投递策略。
- 恢复已到期的 `suppressed`、`snoozed` 和崩溃前已认领的 `claimed` 事件。
- 禁用或软删除后不恢复旧事件。
- 桌面运行时每条提醒只保留恢复窗口中最新一次新计划，避免休眠或时间跳变后批量弹窗。
- 一次性提醒按策略在 24 小时内恢复，超出窗口标记 `missed` 而不主动展示。

### 2.2 候选级策略上下文

调度器新增 `RuntimeContextSource`，每个候选可独立读取：

- 当前 reminder revision 与启用状态。
- 单条暂停结束时间。
- 全局暂停结束时间。
- 锁屏/休眠可用状态。
- 唤醒 30 秒冷却。
- Windows 前台全屏探测结果。

旧的固定 `RuntimeContext` 调用方式仍保留，现有领域和应用测试无需改写。

### 2.3 后台调度循环

- 启动时执行 24 小时恢复窗口对账，但循环提醒不会批量补发。
- 平时精确等待下一条固定时间、一次性或延期事件。
- 30 秒兜底对账用于弥补少量系统事件丢失，不依赖高频轮询。
- 配置变更通过内部信号立即唤醒调度器。
- 锁屏/休眠后标记会话不可用；唤醒/解锁先结算恢复事件，再进入 30 秒冷却。
- 系统时间回拨时保留重叠查询窗口，最终由 SQLite 唯一键防止重复展示。

### 2.4 桌面投递与规则恢复

- 系统通知标题和正文来自本地提醒配置。
- 投递前数据库已保存 `Deliver` 决策；成功后进入 `presented`，失败进入 `delivery_failed`。
- 应用重启后，提醒列表从 `schedule_rules` 恢复规则摘要和下一次触发时间。
- 一次性提醒使用当地日期时间 + IANA 时区创建；过去时间和 DST 不存在时刻会被前后端共同拒绝。
- 损坏或未知规则不会导致整个提醒列表消失。

## 3. 恢复与防重证据

自动化测试已覆盖：

- 同一固定时间候选重复 reconcile 只投递一次。
- 使用同一个 SQLite 文件模拟进程重启后，相同候选命中唯一键且不再次投递。
- 崩溃发生在“已认领、未作策略决定”之间时，`claimed` 事件会在启动后恢复。
- 崩溃发生在 `delivering` 后时不自动重弹，继续遵循“最多展示一次”的产品取舍。
- 策略上下文查询失败发生在认领前，不会制造新的孤立 `claimed` 事件。
- 多个并发认领者中只有一个能成功。
- due deferred 使用原 occurrence identity 恢复，不会创建第二条事件。
