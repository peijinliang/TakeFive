# 摸个鱼 · TakeFive 阶段 0 技术预研报告

**报告版本：** V0.1  
**日期：** 2026-07-14  
**验证平台：** Windows 11 x64，Build 26200  
**对应架构：** `Tauri 2 + Rust + React/TypeScript + SQLite（后续阶段接入）`

---

## 1. 预研结论

Tauri 2 方案在 Windows 上具备继续进入阶段 1 的技术基础，结论为 **有条件通过**。

已证明：

- Rust 主进程、系统 WebView、React 界面和 IPC 可以形成完整桌面闭环。
- 固定时间和一次性提醒可以在独立领域层中计算，不依赖界面或真实等待。
- Windows 可以在不读取输入内容的前提下获取系统空闲时长。
- 前台全屏启发式判断、多显示器数量、系统通知、托盘和单实例均可接入。
- Windows 电源广播与 WTS 会话消息监听可以在隐藏消息窗口中运行。
- 主窗口销毁后仅保留 Rust/托盘核心，可把后台私有内存降到约 5.5 MB。
- 再次启动应用会唤起原实例并重建主窗口，不产生第二个常驻实例。

尚未证明：

- Windows 真实休眠/唤醒、锁屏/解锁后的事件顺序与丢失情况，需要人工演练。
- macOS 原生能力必须在 macOS 真机验证，不能由 Windows 构建结果推断。
- SQLite 事务、事件唯一键与启动对账属于阶段 1，不在本次最小原型中。

## 3. 已实现原型

### 3.1 工程边界

```text
TakeFive/
  apps/desktop/          React + Tauri 桌面应用
  crates/domain/         纯 Rust 时间与规则领域层
  docs/                  架构与预研文档
  artifacts/             本次运行截图
```

领域 crate 不依赖 Tauri、Win32 和界面代码，后续可以直接被调度器、数据库集成测试和 macOS 桌面壳复用。

### 3.2 时间模型

已实现：

- `Clock` 抽象。
- `SystemClock`。
- 可设置、可推进的 `FakeClock`。
- 工作日固定时间规则。
- 一次性绝对时间规则。
- 稳定事件唯一键。
- DST 不存在时间顺延到第一个有效时刻。
- DST 重复时间只选择第一次，防止同一本地计划触发两次。

### 3.3 Windows 能力

| 能力 | 当前状态 | 实现方式 |
| --- | --- | --- |
| 系统空闲时长 | 已验证 | `GetLastInputInfo` |

| 前台全屏判断 | 已验证基础路径 | 前台窗口边界与活动显示器边界比较 |

全屏判断只能作为降低打扰的启发式信号，不能升级为安全或权限判断。

## 5. 资源基线

测试为当前预研界面和 WebView2 145，未接入 SQLite、图表和完整业务页面。

| 状态 | 私有内存 | 工作集 | 稳定态 CPU |
| --- | ---: | ---: | ---: |
| 主窗口打开 | 约 212.8 MB | 约 407.6 MB | 整机归一化约 0.06% |
| 主窗口关闭、托盘常驻 | 约 5.5 MB | 约 27.8 MB | 15 秒采样为 0% |

说明：

- 工作集包含 WebView2 共享页，不能直接等同应用独占内存，因此同时记录私有内存。
- 单纯隐藏窗口不会释放 WebView2，私有内存仍约 211 MB。
- 当前实现改为关闭时销毁主 WebView，后台只保留 Rust 核心与托盘。
- 打开窗口时仍超过需求建议的 150 MB。若 150 MB 是“任何状态下”的硬门槛，需要增加 Flutter 或 Windows 原生 UI 的对照原型；若指标指后台常驻状态，当前方案满足目标且余量充足。

## 7. 风险与决策

| 风险 | 当前判断 | 后续动作 |
| --- | --- | --- |
| 打开窗口私有内存高于 150 MB | 中 | 明确指标口径；阶段 1 持续监测，必要时做 Flutter 对照 |
| 销毁窗口后 UI 状态丢失 | 可控 | 业务事实放 SQLite，临时表单关闭前提示或保存草稿 |
| Win32 生命周期消息可能因设备差异丢失 | 中 | 在笔记本、台式机、Modern Standby 上分别演练 |
| 全屏检测有误判 | 已知边界 | 保持可配置策略，不申请高风险权限 |
| macOS 能力尚未验证 | 高 | 阶段 1 前安排 macOS 真机与签名环境 |
| WebView2 版本差异 | 中 | Windows 10/11 多版本和旧 Runtime 回归 |

### 暂定架构决策

继续采用 Tauri 2 进入阶段 1，但附带两个条件：

1. 把“后台常驻内存”与“主窗口打开内存”拆成两个指标。
2. 在 macOS 原生能力预研完成前，不宣布跨平台架构最终冻结。

## 8. 待人工演练

Windows 当前可以执行以下人工验证：

1. 保持预研窗口打开，锁定并解锁系统，观察“本次验证记录”是否出现 lock/unlock。
2. 保持应用运行，使电脑进入休眠后唤醒，观察 sleep/wake 顺序。
3. 打开全屏视频或演示，确认“前台全屏判断”变化。
4. 点击“申请通知权限”和“发送测试通知”，确认通知中心展示。
5. 关闭主窗口后从托盘重新打开，确认窗口重建。

这些动作会改变当前桌面会话，因此没有在自动化过程中强制执行。

## 10. 阶段 1 入口建议

阶段 1 优先顺序保持为：

1. SQLite schema、迁移和 Repository。
2. Occurrence 事件唯一键与事务认领。
3. 固定时间、间隔循环和一次性规则统一接口。
4. 启动、时间变化、锁屏和唤醒对账器。
5. 完成、跳过、延后状态机。
6. 用 Fake Platform 扩展故障与生命周期测试。

在事件模型与恢复链路完成前，不进入完整页面、主题和统计开发。
